## 1\. AI 行为红线（最高优先级，全局强制执行）

### 1\.1 绝对禁止行为

- 禁止修改、删除、注释与当前需求无关的业务代码

- 禁止擅自整体重构模块、调整项目目录结构

- 无明确指令时，禁止删除已有业务逻辑、变量、函数、接口

- 禁止私自修改对外 API 协议、请求参数、响应结构、路由地址

- 禁止无指令变更数据表结构、字段类型、索引、数据约束

- 禁止编写功能重复的方法、工具类、接口

- 禁止私自扩大需求范围，新增需求以外的功能逻辑

### 1\.2 强制执行原则

- 最小化修改原则：仅改动当前需求对应的代码，严格控制改动范围

- 业务逻辑、边界条件、依赖关系存疑时必须主动提问，禁止自行脑补逻辑

- 禁止在无明确指令的情况下“顺手优化”历史代码、格式、命名

- 新增代码必须贴合项目现有技术风格与架构，禁止私自引入未评估的新技术、新框架

### 1\.3 AI 事实约束原则（防幻觉编码 · 强制）

1. 未提供数据表结构、字段说明时，**禁止自行猜测数据库字段**

2. 未提供 API 文档、参数定义时，**禁止自行编造接口入参、出参结构**

3. 未明确业务规则、流程逻辑时，**禁止擅自补全、改写业务逻辑**

4. 所有不确定的字段、参数、规则、逻辑，二选一执行：

    - 方式一：主动提问确认后再进行编码

    - 方式二：代码/注释中明确标记 `ASSUMPTION：此处为假设内容，需业务核对`

## 2\. 架构分层规范

### 2\.1 Controller 控制层

#### 职责

- 接收 HTTP 请求，解析请求入参

- 统一参数校验、身份认证、权限校验

- 统一接口响应封装、异常捕获与转发

- 仅调用 Service 层，不直接调用 Repository、外部服务

#### 禁止

- 编写核心业务逻辑

- 直接操作数据库、执行 SQL 语句

- 手动拼接业务返回数据

### 2\.2 Service 业务层

#### 职责

- 实现核心业务逻辑、业务流程编排

- 统一管理事务、缓存读写、外部服务调用

- 调用 Repository 层完成数据存取

#### 禁止

- 直接处理 HTTP 请求头、响应头、状态码

- 直接返回数据库原始实体，必须做数据脱敏与结构封装

- 多层嵌套调用其他 Service，规避循环依赖

#### Service 通信规则

禁止 Service 之间互相直接注入调用，跨业务模块通信统一通过**事件总线**或**公共基础 Service** 解耦。

### 2\.3 Repository 数据访问层

#### 职责

- 封装数据库通用 CRUD、分页、联表查询操作

- 仅负责纯数据读写逻辑

#### 禁止

- 编写业务判断、流程分支等业务逻辑

- 调用外部 API、中间件、其他业务服务

### 2\.4 模块自治原则

1. 每个业务 `module` 独立闭环，包含专属 Controller、Service、Repository、类型定义

2. 禁止跨模块直接调用对方 Repository，数据交互必须通过对方对外暴露 Service

3. 模块仅对外暴露能力接口，内部实现完全隔离，适配单体、微服务、多租户架构演进

## 3\. 代码风格与格式规范

### 3\.1 全局命名规范

|类型|命名风格|示例|
|---|---|---|
|普通变量 / 函数 / 方法|camelCase 小驼峰|`getUserById`|
|类 / 接口 / 枚举 / 自定义类型|PascalCase 大驼峰|`UserService`|
|全局静态常量 / 枚举常量|UPPER\_SNAKE\_CASE 全大写下划线|`MAX_QUERY_LIMIT`|
|源码文件|kebab\-case 短横线|`user-service.ts`|
|数据库表 / 字段|snake\_case 下划线|`user_id`|

### 3\.2 代码格式规范

- 缩进：统一使用 **2 个空格**，严禁使用 Tab

- 单行最大字符长度：120 字符，超长语句强制换行

- 字符串：统一使用**单引号**

- 语句结尾：默认不加分号（各技术栈可按自身习惯微调）

- 花括号：左花括号 `{` 紧跟代码同行，不单独换行

- 导入/引用语句：按「第三方包 → 项目公共模块 → 本业务模块」分组，组之间空一行

## 4\. 注释规范（核心强制）

### 4\.1 通用规则

- 所有注释**必须使用中文**，仅第三方库原生注释可保留英文

- 注释重点解释**业务背景、设计原因、特殊限制**，不重复复述代码功能

- 禁止无意义废话注释、语义重复注释、过度堆砌注释

### 4\.2 无需额外注释场景

以下代码语义自明，无需添加注释，避免降低代码可读性：

1. 语义清晰的变量赋值：`const isAdmin = user.role === 'admin'`

2. 编程语言原生标准库方法调用：`arr.filter()`、`arr.map()`

3. 逻辑直观的条件判断：`if (user.age >= 18)`

### 4\.3 文件头注释（所有源码文件必填）

```TypeScript
/**
 * @文件描述 模块/文件核心功能说明
 * @作者 姓名
 * @创建时间 YYYY-MM-DD
 * @最后修改 YYYY-MM-DD
 */
```

### 4\.4 函数/方法 JSDoc 注释（公共、异步、对外暴露函数必填）

```TypeScript
/**
 * 根据用户ID获取用户基础信息
 * @param userId - 用户唯一标识
 * @returns 脱敏后的用户信息对象
 * @throws 数据库连接异常、数据不存在业务异常
 */
async function getUserById(userId: string) {}
```

### 4\.5 行内注释规则

- 魔法数字、特殊配置、逻辑拦截点必须添加行内注释

- 简短注释可写在代码右侧，长注释统一写在代码上方

✅ 正确示例

```TypeScript
const TOKEN_EXPIRE = 86400 // Token 有效期 1天，单位：秒
// 用户未登录直接终止流程，跳过后续鉴权逻辑
if (!user) return null
```

❌ 禁止示例

```TypeScript
i++ // i 自增（废话注释，禁用）
// add num（英文注释，禁用）
```

### 4\.6 标记注释统一格式

格式：`// 标签(姓名 日期): 问题/待办说明`

```TypeScript
// TODO(张三 2026-06-15): 后续对接短信接口，补充验证码下发逻辑
// FIXME(张三 2026-06-15): 并发场景存在竞态，需增加分布式锁
// HACK(张三 2026-06-15): 临时兼容第三方SDK缺陷，官方修复后移除
```

### 4\.7 强制添加注释场景

以下代码片段**必须补充注释**：

- 正则表达式（说明匹配规则、分组含义）

- 位运算（说明运算意图、等价写法）

- 魔法数字（优先提取为常量并注释）

- 代码嵌套层级 ≥3 层的逻辑块

- 异步、并发、限流、排队控制逻辑

- 自定义算法、特殊数据处理逻辑

## 6\. 函数与模块设计规范

- 推荐单个函数代码行数 ≤ 50 行，超出建议拆分；因业务复杂性无法拆分时，**必须在文件头部添加注释说明原因**

- 推荐单个文件代码行数 ≤ 300 行，超出建议拆分；特殊聚合类文件可例外，**必须标注例外原因**

- 函数入参个数推荐 ≤ 3 个，参数过多统一使用对象传参

- `if/for` 代码嵌套层级推荐 ≤ 3 层，深层嵌套优先重构

- 严格遵循单一职责原则，一个函数仅实现一项功能

- 禁止直接使用魔法数字，所有固定数值必须定义为具名常量

## 7\. TypeScript 类型规范

> 本章节为 **TypeScript/JavaScript 专属规范**，其他技术栈可忽略

- 优先全面避免使用 `any`；对接老旧第三方 SDK、无类型声明的包时，允许局部使用 `any`，**必须在当前行/代码块上方注释说明使用原因**

- 普通对象结构优先使用 `interface`；联合类型、工具类型使用 `type`

- 所有公共 API 的入参、返回值必须完整定义类型

- 枚举命名使用 PascalCase，**每个枚举成员必须添加中文注释说明含义**

## 8\. API 接口规范

### 8\.1 基础规则

- 遵循 RESTful 设计风格，资源名称使用名词复数

- 接口统一版本前缀：`/api/v1/`

### 8\.2 分页参数（全局统一）

```TypeScript
page: number     // 页码，从 1 开始
pageSize: number // 每页数据条数
```

### 8\.3 全局统一响应结构

```TypeScript
interface ApiResponse<T> {
  code: number    // 业务错误码，0 代表请求成功
  message: string // 响应描述文案
  data: T | null  // 业务数据，无数据返回 null
}
```

### 8\.4 接口幂等性规范

1. 强制要求幂等接口：支付接口、下单类 POST 接口、数据状态变更接口

2. 实现方案：客户端传入全局唯一 `requestId`，服务端基于该标识做重复请求去重

3. 保证同一 `requestId` 多次请求，返回结果一致，不产生重复业务数据

### 8\.5 API 向后兼容规范

1. 已上线接口**禁止删除原有字段**

2. 禁止修改已有字段的含义、数据类型

3. 接口新增字段必须设置为可选（optional），兼容旧版客户端

4. 接口发生不兼容变更时，必须升级版本（`/api/v1` → `/api/v2`）

### 8\.6 API 契约隔离规范

1. 前后端通过 DTO/数据模型对齐接口契约

2. 禁止直接将数据库实体对外暴露

3. 所有对外响应数据必须经过 DTO 转换、脱敏后返回

## 9\. 数据库操作规范

- 表名、字段统一使用 `snake_case` 下划线命名

- 禁止编写裸 SQL，统一通过 ORM / Repository 层操作

- 索引命名规范：单字段 `idx_表名_字段名`；多字段 `idx_表名_字段1_字段2`

- 所有数据表强制包含通用基础字段：

    - `created_at`：数据创建时间

    - `updated_at`：数据更新时间

    - `created_by`：数据创建人

    - `deleted_at`：软删除字段，`null` 代表未删除，存在时间戳代表已删除

### 补充：软删除\+唯一索引约束

存在 `deleted_at` 软删除的表，**唯一索引必须包含 ****`deleted_at`**** 字段**，避免旧数据软删除后，同名新数据无法插入。

### 9\.5 数据库 DDL 变更流程规范

1. 数据表结构、字段、索引、约束变更，**必须提供数据库迁移文件（migration）**

2. 变更文档必须明确：是否支持回滚、是否会锁表、是否影响线上存量数据

3. 生产环境**禁止手动直接执行 DDL 语句**，所有结构变更通过迁移脚本上线

## 11\. 缓存规范

### 11\.1 Key 命名格式

格式：`模块名:实体名:唯一标识`
示例：

```Plaintext
user:profile:1001
order:info:20260615001
```

### 11\.2 过期时间规则

- 禁止硬编码过期时间，统一抽取为全局常量

- 热点数据必须设置过期时间，禁止永久缓存

### 11\.3 缓存与数据库一致性策略

全局统一采用 **Cache\-Aside（旁路缓存）** 模式：

1. 读流程：先查缓存 → 未命中再查询数据库 → 结果回写缓存

2. 写流程：先更新数据库 → 再删除对应缓存

3. 禁止先删缓存再更新数据库，规避并发脏读问题

### 11\.4 分布式锁规范

资金操作、唯一凭证生成等高并发竞态场景，**必须使用 Redis 分布式锁（Redlock 等方案）**，禁止仅依赖数据库实现并发控制。

### 11\.5 附加约束

- 数据更新/删除时，同步清理对应缓存

- 核心业务数据需做缓存穿透、缓存击穿防护

## 12\. 日志规范

### 12\.1 基础规则

- 禁止直接使用 `console.log` 原生打印，统一使用项目封装的日志工具

- 日志分级：`info`（正常日志）、`warn`（警告日志）、`error`（错误日志）

### 12\.2 日志必填字段

所有业务日志必须包含：

- traceId：链路追踪 ID

- userId：操作用户 ID

- module：所属业务模块

- params：脱敏后的核心入参

### 12\.3 安全禁令

日志中**严禁打印**明文密码、手机号、身份证、Token、密钥等敏感信息。

## 13\. 错误处理规范

> TypeScript/JavaScript 技术栈统一使用 `try/catch` 机制，其他语言参考专属规范

### 13\.1 基础规则

- 所有异步逻辑、IO 操作必须捕获异常

- **禁止空 catch 块**，不允许静默吞噬错误

- 自定义业务错误必须原生继承 `Error` 基类

- 捕获异常后，必须记录完整日志（堆栈、上下文、入参）

- Controller 层统一拦截异常，按照全局 API 格式返回响应

### 13\.2 错误码定义规范

1. 错误码统一集中维护在 `src/constants/error-code.ts`

2. 分段规则：

    - `0`：接口请求成功

    - `1000 - 1999`：通用系统错误（1001 参数非法、1002 未登录）

    - `2000 - 2999`：用户模块业务错误

    - `3000 - 3999`：订单模块业务错误

    - 后续模块按区间依次顺延分配

## 14\. 安全开发规范

### 14\.1 通用安全规则

- 密钥、数据库账号、密码、Token 等敏感信息禁止硬编码，统一通过环境变量注入

- 所有外部入参必须做合法性校验，防范 SQL 注入、非法参数攻击

- 项目依赖版本必须锁定，保留 `package-lock.json` / `yarn.lock` / `go.sum` 等锁定文件

- 接口返回数据按需脱敏，禁止返回用户隐私字段

### 14\.2 外部依赖降级与容错规范

1. 调用第三方 HTTP 接口、Redis、MQ 等外部依赖，必须设计降级/兜底策略

2. 缓存服务异常，不可阻断主业务流程

3. 消息队列消费失败，必须支持重试机制

4. 区分链路等级：核心依赖失败抛出异常；非核心依赖失败执行降级逻辑

### 14\.3 AI 业务专属安全规则

接口接收用户输入并转发至大模型 LLM 时，**必须对输入内容做清洗、转义、过滤**，防范 Prompt 提示词注入攻击。

## 15\. 性能规范

### 15\.1 禁止行为

- 禁止循环内执行数据库查询（杜绝 N\+1 问题）

- 禁止无限制全表查询、不带分页的大数据量查询

### 15\.2 强制优化要求

- 批量数据场景优先使用批量查询、批量写入

- 高频查询字段必须建立合理数据库索引

- 大数据列表查询强制分页，并设置最大查询条数限制

## 16\. 测试编码规范（强制）

### 16\.1 基础要求

- 业务代码、工具函数、接口**必须配套编写测试用例**

- 核心业务逻辑测试覆盖率 ≥ 80%

- 测试文件命名：`xxx.test.ts` / `xxx.spec.ts`，与业务文件同目录

- Mock 模拟数据统一存放至 `__mocks__` 目录

### 16\.2 用例命名规则

统一格式：`应该_预期结果_当_触发条件 [场景类型]`
**示例**：

1. `应该返回用户信息_当传入合法用户ID时 [正常场景]`

2. `应该返回空数据_当用户ID不存在时 [边界场景]`

3. `应该抛出参数错误异常_当传入空ID时 [异常场景]`

### 16\.3 强制覆盖场景

每个功能必须覆盖三类场景：

1. 正常场景：合法入参、正向业务流程

2. 边界场景：参数极值、空数据、分页边界

3. 异常场景：非法入参、依赖报错、业务异常

### 16\.4 标准测试结构（三段式）

1. Arrange：前置准备、初始化数据、Mock 依赖

2. Act：执行待测试逻辑

3. Assert：精准断言结果、状态、数据变更
