## 1. 全局通用规范

### 1.1 API 前缀

所有业务接口统一前缀：`/api/v1`

```text
GET  /api/v1/health                          # 健康检查
GET  /api/v1/pets/:pet_id/medical-episodes/current  # 医疗
POST /api/v1/dog-walk/opportunities/detect          # 代遛
```

### 1.2 统一响应格式

所有接口返回 JSON 格式，遵循以下结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": null
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 业务错误码，`0` 代表成功 |
| `message` | string | 响应描述 |
| `data` | any | 业务数据，可为 `null` |

分页接口 `data` 内嵌分页结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "list": [],
    "total": 100,
    "pageNum": 1,
    "pageSize": 20
  }
}
```

### 1.3 ID 生成规则

全局统一使用 **ULID** 算法，带业务前缀。数据库字段类型统一 `varchar(64)`。

| 前缀 | 业务实体 | 示例 |
|------|---------|------|
| `ep_` | 医疗疗程 | `ep_01JZYQ4M8PABCDEFGHJKLMNPQR` |
| `task_` | 护理任务 | `task_01JZYQ5BXXXXXYZ...` |
| `act_` | 任务操作记录 | `act_01JZYQ6C...` |
| `sum_` | 医疗摘要 | `sum_01JZYQ7D...` |
| `auth_` | 医疗授权 | `auth_01JZYQ8E...` |
| `opp_` | 代遛需求 | `opp_01JZYQ9F...` |
| `plan_` | 代遛方案 | `plan_01JZYQ0G...` |
| `order_` | 代遛订单 | `order_01JZYQ1H...` |
| `event_` | 实时事件 | `event_01JZYQ2I...` |
| `report_` | 服务报告 | `report_01JZYQ3J...` |
| `evt_` | Kafka 事件 ID | `evt_01JZYQ4K...` |

### 1.4 全局请求 Header

所有业务接口（`/health` 除外）必须携带以下 Header：

| Header | 必选 | 说明 |
|--------|------|------|
| `Authorization` | ✅ | 身份凭证 Token |
| `X-Trace-Id` | ✅ | 全链路追踪 ID，ULID 格式（26 或 32 字符），不存在则自动生成 |
| `X-Request-Source` | ✅ | 请求来源：`client` / `scheduler` / `third_party` |
| `X-Timestamp` | ✅ | Unix 秒级时间戳（防重放校验） |
| `X-Signature` | ✅ | HMAC-SHA256 签名（防重放校验） |
| `Idempotency-Key` | POST/PUT 写操作 | 幂等键，相同 key 的重复请求返回首次结果 |
| `X-App-Version` | - | 客户端版本号 |

### 1.5 分页规范

所有列表类接口统一使用查询参数传递分页：

| 参数 | 类型 | 默认值 | 最大值 | 说明 |
|------|------|--------|--------|------|
| `pageNum` | int | 1 | - | 页码，从 1 开始 |
| `pageSize` | int | 20 | 50 | 每页条数，超出自动截断 |

分页中间件自动注入，handler 通过 context 获取 `pageNum` / `pageSize`。

### 1.6 时间格式

- 所有时间字段统一使用 **UTC+8（东八区）**
- 格式：`yyyy-MM-ddTHH:mm:ss+08:00`（例如 `2026-06-16T10:30:00+08:00`）
- 数据库存储：`timestamptz`

## 2. 中间件与安全

中间件执行顺序（Gate 模式）：

```
客户端请求
  │
  ▼
┌─────────────────────┐
│ IP 拦截             │  黑名单/白名单模式，支持 CIDR
│ (IPBlocker)         │  可选，通过配置控制启用
├─────────────────────┤
│ 限流                │  基于 Redis 滑动窗口
│ (RateLimiter)       │  默认 100 次/分钟/IP
├─────────────────────┤
│ 链路追踪            │  透传/生成 X-Trace-Id
│ (GinTraceMiddleware)│  注入 context，日志携带
├─────────────────────┤
│ Header 校验          │  Authorization / X-Request-Source 必填
│ (GinHeaderValidation)│  /health 跳过
├─────────────────────┤
│ 分页注入            │  解析 pageNum/pageSize
│ (GinPagination)     │  注入 context
├─────────────────────┤
│ 认证                │  JWT Token 解析及用户身份注入
│ (GinAuthMiddleware) │  返回 RequestContext 含 UserID
├─────────────────────┤
│ 幂等校验            │  POST/PUT/PATCH 强制 Idempotency-Key
│ (Idempotency)       │  2xx 响应缓存 24h，重复请求返回缓存
├─────────────────────┤
│ 防重放              │  X-Timestamp ±5min / HMAC-SHA256 签名
│ (ReplayMiddleware)  │  密钥从环境变量 REPLAY_SECRET_KEY 读取
└─────────────────────┘
  │
  ▼
  Handler
```

### 限流配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `MaxRequests` | 100 | 时间窗口内最大请求数 |
| `Window` | 1 minute | 滑动窗口大小 |
| `KeyFunc` | 按 ClientIP | 限流维度 |

### 防重放签名算法

```
签名内容 = requestBody + timestamp(字符串)
签名算法 = HMAC-SHA256(签名内容, REPLAY_SECRET_KEY)
```

### 熔断规则（第三方接口）

- 连续 10 次失败 → 熔断 5 分钟
- 熔断期返回兜底数据
- 超时时间：3s

### 4.3 生成医嘱任务草稿

```
POST /api/v1/medical-episodes/:episode_id/care-tasks/draft
```

**路径参数**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `episode_id` | string | ✅ | 疗程 ID |

**请求体**

```json
{
  "tasks": [
    {
      "task_type": "medication",
      "title": "喂药：阿莫西林 250mg",
      "instruction": "每日两次，每次一片，饭后服用",
      "due_at": "2026-06-17T08:00:00+08:00",
      "risk_level": "P1",
      "lock_fields": ["medication_name", "dosage"]
    }
  ]
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `tasks[].task_type` | string | ✅ | 任务类型（medication / care_measure / observation / exercise 等） |
| `tasks[].title` | string | ✅ | 任务标题 |
| `tasks[].instruction` | string | - | 执行说明 |
| `tasks[].due_at` | string | - | 截止时间（UTC+8） |
| `tasks[].risk_level` | string | - | 风险等级（P0 / P1 / P2） |
| `tasks[].lock_fields` | string[] | - | 需要锁定的字段列表（AI 不可修改） |

**状态机约束**

- 仅 `EpisodeCreated` 状态的疗程允许生成草稿
- 成功后疗程状态推进至 `InstructionDrafted`

**响应 `201 Created`**

```json
{
  "code": 0,
  "message": "任务草稿生成成功",
  "data": {
    "episode_id": "ep_01JZYQ...",
    "tasks": []
  }
}
```

**错误码**

| 码值 | 含义 |
|------|------|
| `2001` | episode_id 为空 |
| `2001` | 查询疗程失败 |
| `2001` | 疗程不存在（404） |
| `2002` | 当前状态不允许生成任务草稿 |
| `2002` | 请求参数无效 |
| `2001` | 保存任务草稿失败 |
| `2001` | 更新疗程状态失败 |

```
  │  POST /opportunities/:id/allow-plan

```

```
                                                      ▼

                                            ▼                   ▼
```

## 8. LLM 输出强制 Schema

所有 LLM 输出必须遵循以下结构，后端强制校验：

```json
{
  "intent": "describe the user intent",
  "action": "execute_tool | notice | deny",
  "risk_level": "P0 | P1 | P2",
  "need_confirm": true,
  "tool_calls": []
}
```

### 全局运行阈值

| 配置项 | 数值 | 说明 |
|--------|------|------|
| 单次会话最大工具调用数 | 5 | `MaxToolCallsPerSession` |
| 工具嵌套最大递归深度 | 3 | `MaxToolNestingDepth` |
| LLM 输入 Token 上限 | 8000 | `MaxLLMInputTokens` |
| LLM 输出 Token 上限 | 1500 | `MaxLLMOutputTokens` |
| Tokens 日预算 | 100,000 | `DailyTokenBudget` |
| Tokens 月预算 | 2,000,000 | `MonthlyTokenBudget` |
| 单用户最大消耗 | 5,000 | `MaxCostPerUser` |
| LLM 降级重试次数 | 1 | `LLMRetryCount` |
| 第三方接口超时 | 3s | `ThirdPartyTimeoutMs` |
| 熔断阈值（连续失败） | 10 | `ThirdPartyCircuitBreakerThreshold` |
| 熔断持续时间 | 5min | `ThirdPartyCircuitBreakerDurationSec` |
| 定时任务超时 | 10s | `ScheduledTaskTimeoutSec` |
| 优雅停机等待 | 30s | `GracefulShutdownTimeoutSec` |
