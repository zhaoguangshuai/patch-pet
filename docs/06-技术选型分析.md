# 06 帕奇宠 Agent 后端技术选型分析

## 一、为什么使用 Go + Python 双语言架构

### 1.1 误区澄清：这不是一个"AI 项目"

帕奇宠本质上是一个**带 AI 能力的后端业务系统**，而不是一个 AI 应用。

从代码分布来看：

```
internal/         18 个包，绝大多数是非 AI
├── agent/        Agent 编排（Go 控制逻辑）
├── medical/      医疗居家 Agent 状态机
├── dogwalk/      代遛狗 Agent 状态机
├── auth/         身份认证 & 权限
├── middleware/    HTTP 中间件（限流/幂等/防重放/追踪/分页）
├── database/     数据库连接 & 迁移 & 通用 Repository
├── eventbus/     Kafka 事件总线
├── workflow/     Saga 分布式事务
├── audit/        全链路审计日志
├── lifeflow/     生命流事件服务
├── policy/       策略引擎（权限 & 风控）
├── memory/       三层记忆管理
├── runtime/      Agent Runtime 执行引擎
├── tools/        工具注册中心
├── health/       健康检查
├── hitl/         人机介入
├── rag/          RAG 客户端（仅接口封装）
└── compliance/   合规 & 数据留存

ai-service/       仅 ~8 个源文件，纯 AI 职责
├── llm/          LLM 推理调用
├── rag/          RAG 向量化 & 检索
├── prompt/       Prompt 管理
├── eval/         AI 自动化评测
├── api/          gRPC 服务端 + Kafka 消费者
└── config/       配置管理
```

**80%+ 的代码是传统的后端基础设施（API、数据库、中间件、状态机、事件总线），只有不到 20% 是 AI 相关代码。** 这一比例直接决定了语言选型的方向。

### 1.2 与产品原型的关系

产品原型（帕奇宠 C 端一期低保真原型 V2.1）共定义了 **23 个场景**（编号 00～FF），涉及家庭管理、设备绑定、生命流、医疗、代遛、口粮推荐、数字化形象等多个领域。当前一期开发仅以下 2 个模块实现为带状态机的 Agent：

| 原型编号 | 场景名称 | 实现模块 | 类型 |
|---------|---------|---------|------|
| **16** | 医疗·居家任务 | `internal/medical/` | 状态机 Agent（P0 高危） |
| **19** | 代遛狗闭环 | `internal/dogwalk/` | 状态机 Agent（P1 中高危） |
| 其余 21 个场景 | — | 待规划中 | 待定 |

其余模块（家庭管理、设备管理、数字化形象、口粮推荐等）**当前尚未进入研发阶段**。本文档的技术选型分析以已落地的 16、19 号模块为基础，同时为后续模块保留架构扩展空间。

各原型模块与实现模式的关系预计如下：

```
┌──────────────────────────────────────────────────────────┐
│ 原型场景 × 实现模式映射（预期）                            │
├──────────────────────────────────────────────────────────┤
│ 状态机 Agent 模块（类似 16、19）：                          │
│   #17 医疗·机构托管（状态跟踪 + 三方回传）                   │
│   #18 ICU 特殊状态（监护回传 + 机构对接）                    │
│   #20 口粮推荐闭环（推荐 → 订单 → 监理闭环）                 │
│                                                          │
│ LLM 增强的 CRUD + 视图模块：                                │
│   #01-#03 家庭/设备/宠物建档（标准 CURD）                   │
│   #09-#13 生命流系列（数据聚合 + LAKI 摘要）                 │
│   #14 走失与项圈（设备数据 + 实时定位推送）                   │
│   #15 饮食与共享设备（数据看板 + 趋势分析）                   │
│                                                          │
│ AI 内容生成模块：                                           │
│   #04 数字化主页（虚拟空间 + 互动）                          │
│   #06 性格 DNA（标签 + 视频/叫声 → 风格定制）                │
│   #08 表情包制作（照片 + 文案 → 生成）                      │
│   #07 IP 实体化（设计 → 商品预览 → 跳转渠道）                │
└──────────────────────────────────────────────────────────┘
```

这种映射关系意味着：**只有涉及「多步骤、跨系统、需要状态持久化」的业务才需要设计为状态机 Agent**；纯粹的展示、内容生成或单次交互模块走标准 CRUD + LLM 增强即可，不应盲目「Agent 化」。

### 1.3 Go 主服务的不可替代性

Go 被选为主语言，原因如下：

| 需求 | Go 的优势 |
|------|----------|
| **高并发** — 医疗 P0 系统需同时处理大量设备上报、定时任务、用户请求 | Goroutine 栈仅 4KB，可轻松支撑数万并发；Python 协程开销大得多，且 GIL 限制 CPU 密集型并行 |
| **中间件链性能** — 每个请求经过 auth → ratelimit → idempotency → replay → gateway 等多层中间件 | Go 编译为原生代码，中间件链开销在 ns 级；Python 每层中间件都有显著的解释器开销 |
| **类型安全** — 医疗场景 P0 高危，状态转换、数据库操作不允许类型错误 | Go 编译期捕获类型错误；Python 需要运行时才能发现 None 访问、类型不匹配等问题 |
| **部署简洁** — 单二进制部署，适合 K8s 容器化 | `go build` 输出 15MB 单文件；Python 需要携带解释器 + 虚拟环境 + 依赖包 |
| **可观测性** — 结构化日志（zap）、指标、链路追踪 | Go 的日志库（zap/zerolog）以零分配闻名；Python 日志在高速场景下有显著性能开销 |

### 1.4 Python AI 服务的不可替代性

AI 部分使用 Python 的原因：

| 需求 | Python 的优势 |
|------|-------------|
| **LLM SDK** — 调用 OpenAI GPT-5、Anthropic Claude 等 | openai、anthropic 等主流 LLM SDK 均为 Python 优先，Go SDK 功能滞后且生态薄弱 |
| **RAG 向量化** — 使用 embedding 模型做语义检索 | sentence-transformers、numpy 等 ML/NLP 生态在 Python 中成熟完善，Go 的 ML 生态基本为零 |
| **Prompt 工程** — 实验性、迭代快的 AI 开发流程 | Python 的动态特性适合快速实验和迭代，REPL 驱动开发效率高 |

### 1.5 为什么不选纯 Java / 纯 Go / 纯 Python

| 方案 | 问题 |
|------|------|
| **纯 Go** | LLM SDK 生态薄弱，RAG 向量化库几乎不存在，AI 开发效率低 |
| **纯 Python** | 高并发场景性能不足，P0 医疗系统的类型安全和审计要求难以满足，部署复杂 |
| **纯 Java** | 项目属于轻量 AI 后端，Java 的 JVM 启动慢、内存占用高的缺点在这个场景被放大，且 AI 生态不如 Python |

### 1.6 双语言架构的实际收益

```
                    ┌────────────┐
                    │ Go 主服务   │  ← 跑在廉价 CPU 实例
                    │ (API/状态机)│     水平扩展处理流量
                    └─────┬──────┘
                          │ gRPC + Kafka
                    ┌─────▼──────┐
                    │ Python AI  │  ← 可独立跑在 GPU 实例
                    │ (LLM/RAG)  │     按调用量扩缩
                    └────────────┘
```

1. **资源隔离**: AI 服务 OOM 不会拖垮 API 网关
2. **独立扩缩**: Go 服务按请求量扩缩，Python 服务按 LLM 调用量扩缩
3. **独立部署**: 更新 Prompt、切换模型不需要重启 Go 服务
4. **团队解耦**: 后端工程师改 Go，AI 工程师改 Python，互不阻塞

### 1.7 小结

> **选型标准：瓶颈是 LLM 调用次数 → Python 合适；瓶颈是请求吞吐和系统复杂度 → Go 更合适。**

这个项目有 18 个 internal 包、6 个中间件、2 个已落地的状态机（16 号医疗 + 19 号代遛）、Saga 事务、审计日志等大量后端基础设施，显然属于后者。Python 只做它最擅长的事——调 LLM 和做向量检索。

## 二、为什么使用纯轻量实现 Agent 而不是 Agent 框架

### 2.1 项目约束评估

在评估是否引入 LangGraph、CrewAI、LangChain 等 Agent 框架之前，先看项目的实际需求：

| 需求 | 本项目实际状况 | 框架能提供的价值 |
|------|--------------|----------------|
| Agent 数量 | 当前 2 个（原型 #16 医疗居家 + #19 代遛狗），未来候选 #17 #18 #20 | 框架擅长 5+ Agent 协作 |
| 流程复杂度 | 有限状态机，路径确定 | 框架擅长动态图结构 |
| 安全约束 | P0 医疗红线，逐行可审计 | 框架的通用安全层无法满足 |
| 部署架构 | Go 编排 + Python 推理异构 | 框架纯 Python，只覆盖一半 |
| 团队规模 | 小团队 | 框架学习成本 > 自研成本 |

### 2.2 框架的通用能力对当前项目是过度抽象

LangGraph 等框架解决的核心问题是**动态图编排**——多个 Agent 互相调用、条件分支、循环直到满足条件。而本项目的流程是高度确定的：

```
医疗 Agent: 医嘱解析 → 任务调度 → 数据汇总 → 诊所对接
代遛 Agent: 需求识别 → 服务商匹配 → 订单履约 → 实时监控
```

用 LangGraph 的 StateGraph 来描述这样一条无分支的线性流程，相当于用 PostgreSQL 存一个只有三行的配置文件——能做，但没意义。

### 2.3 P0 医疗系统的安全约束

```
Agent 禁止直接操作数据库、直连第三方接口
所有工具调用必须经过工具网关 + 策略引擎双重校验
```

这些约束要求 Agent 调用链路的每一层都可审计、可测试、可单独加校验：

```
LLM 意图生成 → 意图路由(Go) → 工具网关(Go) → 策略引擎(Go) → 工具执行
                                                           ↓
                                                    审计日志(Go)
```

引入 LangGraph 后，框架内部的 Agent 循环是黑盒，无法在每一层插入医疗专属的安全检查。

### 2.4 Go + Python 异构架构的约束

Go 负责编排和状态机，Python 只做 LLM 推理。LangGraph 是纯 Python 生态，只能替换 Python 侧的 `llm/client.py`，Go 侧的编排层完全不受影响：

```
Before (当前):
  Go 编排层 → gRPC → Python LLM 推理

After (接入 LangGraph):
  Go 编排层 → gRPC → Python LangGraph Agent → LLM
                     └── 只改这一块
```

这意味着即使未来要接入框架，**改动范围也局限在 `ai-service/` 内部**，Go 侧不需要任何修改。

### 2.5 LAKI 的通用角色：不仅是 Agent，更是全模块对话层

产品原型揭示了一个关键定位：**LAKI 不是特定 Agent 的专属能力，而是贯穿所有模块的通用对话层。**

在原型中，LAKI 以底部输入栏的形式常驻于所有生命流页面和设备管理页面，用户可随时询问任何宠物相关问题。这意味着：

```
┌──────────────────────────────────────────────────────────┐
│                  用户输入（语音/文字）                      │
│                            ↓                             │
│                    LAKI 意图路由                           │
│               ┌──────────┼──────────┐                    │
│               ↓          ↓          ↓                    │
│          状态机 Agent   CRUD 增强  AI 内容生成              │
│          (#16 医疗     (#09 生命流  (#04 数字化形象          │
│           #19 代遛)     摘要)       #06 性格 DNA)          │
│               ↓          ↓          ↓                    │
│               └──────────┼──────────┘                    │
│                          ↓                               │
│                    生命流统一呈现                           │
└──────────────────────────────────────────────────────────┘
```

**架构影响：**

1. **LAKI 路由层必须是通用而非 Agent 专属**——所有请求先经 LAKI 路由，再分发到 Agent / CRUD / 内容生成等不同后端逻辑
2. **Agent 只处理「多步骤 + 前置状态」的复杂流程**（如医疗任务调度、代遛订单履约），单轮问答或数据查询直接由 CRUD + LLM 增强完成
3. **状态机 Agent 不是唯一需要 LLM 的模块**——生命流摘要、每日故事线、性格 DNA 定制都需要 LLM，但不需要状态机

当前 `internal/agent/` 中的编排层已支持通用路由分发，状态机模块（`medical/`、`dogwalk/`）仅注册自己的流程到 LAKI 路由中，非 Agent 流程（如 CRUD 问答）直接由 Go 侧处理后回给 LLM 生成自然语言回复。

### 2.6 轻量实现的实际收益

| 维度 | 当前轻量实现 | 接入框架 |
|------|-------------|---------|
| 依赖数量 | Python 依赖 8 个 | 最少增加 20+ 个（langchain-core, langgraph, langsmith 等） |
| 调试成本 | 100% 自己的代码，报错即定位 | 先排查是框架 bug 还是业务 bug |
| 构建时间 | 秒级 | 分钟级（安装大量依赖） |
| 容器镜像 | 小 | 大（框架依赖数百 MB） |
| 审计合规 | 每行可审计 | 框架内部逻辑为黑盒 |

### 2.7 未来迁移路径

当前的架构设计已经为框架迁移做好了准备：

```
ai-service/
├── llm/client.py          ← 替换为 LangGraph Agent 入口
├── prompt/manager.py      ← 可复用
├── rag/                   ← 可复用
├── api/grpc_server.py     ← 不变（对外接口不变）
└── api/kafka_consumer.py  ← 不变
```

**gRPC 接口是稳定的契约层**，Go 编排层不关心 Python 侧内部用的是裸 LLM 调用还是 LangGraph，只要 gRPC 接口不变，Go 侧零改动。

### 2.8 何时应该引入 Agent 框架

当项目出现以下信号时，考虑引入：

1. **Agent 数量超过 5 个**（对应原型中 #16 #17 #18 #19 #20 全部实现为状态机 Agent），且存在 Agent 间互相调用
2. **流程出现动态分支**，无法预定义状态机路径
3. **需要开箱即用的持久化、断点续跑、人机交互**
4. **团队已有 LangChain/LangGraph 技术积累**
5. **需要 LangSmith 等可观测平台**

### 2.9 小结

> **当前自研不是做不了框架，而是项目约束（医疗安全 > 开发速度，有限状态机 > 动态图，Go + Python 异构）恰好让框架的收益被抵消，成本却被放大。**

轻量实现提供了足够的灵活性，同时在依赖、调试、安全、部署方面都优于引入框架。未来如果需求变化，切换到框架只需要重写 `ai-service/llm/client.py`，Go 编排层完全不动。
